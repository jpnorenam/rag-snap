# Cross-app handoff contract: local UI → Web UI (postMessage)

**Status:** contract only — no implementation yet.
**Audience:** the local UI (`ui/`, served by `ragd` on `http://127.0.0.1:<port>`) and the deployed
Web UI (`https://canonical-req-8605.web.app`).

## Purpose

After a user runs an answer batch in the local UI, a **Collaborate in Web UI** button hands the
just-generated results to the deployed Web UI. Because a deployed HTTPS site cannot fetch back
into the local HTTP daemon (CORS + `SameSite=Strict` loopback cookie + Private Network Access all
block it — see the feasibility notes), the data flow is **reversed**: the local UI is the *sender*
and pushes the payload to the Web UI over `window.postMessage`. No daemon changes, no CORS, no
token leaves the loopback origin.

## Roles and origins

| Role | Origin | Notes |
|---|---|---|
| **Sender** | local UI | `http://127.0.0.1:<port>` or `http://localhost:<port>` — port is OS-assigned and varies per run |
| **Receiver** | Web UI | `https://canonical-req-8605.web.app` — fixed, known ahead of time |

The sender's origin is **not** knowable in advance (random loopback port), which shapes origin
validation below: the receiver validates by *scheme + host allowlist*, not an exact string.

## Constants (both sides hardcode these)

```
WEB_UI_ORIGIN   = "https://canonical-req-8605.web.app"   // exact; sender targets this
PROTOCOL        = "rag-answer-batch-handoff"              // discriminator on every message
PROTOCOL_VERSION = 1                                      // integer; bump on breaking change
```

## Message flow (three steps, receiver-initiated)

The sender opens the receiver in a new tab and then **waits for the receiver to announce it is
ready** before sending — never sends on a timer, because the tab's load time is unknown.

```
Local UI (sender)                         Web UI (receiver)
      │                                          │
      │ 1. window.open(WEB_UI_ORIGIN + path)     │
      │─────────────────────────────────────────▶  (tab loads, mounts, installs listener)
      │                                          │
      │            2. HANDOFF_READY              │
      │  ◀───────────────────────────────────────  postMessage to opener, targetOrigin = "*"
      │     (sender matches by protocol+origin)  │
      │                                          │
      │ 3. HANDOFF_PAYLOAD                        │
      │─────────────────────────────────────────▶  postMessage to the opened window,
      │     targetOrigin = WEB_UI_ORIGIN          │  targetOrigin pinned; receiver validates origin
      │                                          │
      │            4. HANDOFF_ACK (optional)      │
      │  ◀───────────────────────────────────────  confirms load; sender can close/toast
```

Rationale for **receiver-initiated**: only the receiver knows when its listener is installed. A
`READY` ping removes the race where the sender posts before the tab's JS is running. The sender
holds the payload until `READY` arrives (or a timeout elapses — see Timeouts).

## Messages

Every message is a plain JSON-serializable object with these envelope fields:

```
protocol: "rag-answer-batch-handoff"   // MUST equal PROTOCOL; ignore messages without it
version:  1                            // PROTOCOL_VERSION
type:     "HANDOFF_READY" | "HANDOFF_PAYLOAD" | "HANDOFF_ACK" | "HANDOFF_ERROR"
nonce:    string                       // correlates READY→PAYLOAD→ACK for one handoff
```

`nonce` is generated by the **sender** and passed into the opened URL (see step 1) so the receiver
can echo it in `READY`; this lets the sender ignore `READY` messages from unrelated tabs.

### 1. `HANDOFF_READY` — receiver → sender

Sent by the Web UI once its message listener is installed and it is prepared to receive a payload.

```json
{
  "protocol": "rag-answer-batch-handoff",
  "version": 1,
  "type": "HANDOFF_READY",
  "nonce": "<echoed from the URL param the sender opened, or empty if none>"
}
```

- Sent with `event.source.postMessage(msg, "*")` **or**, preferably, targeted at the opener's
  origin if the receiver was told it (it is not, by default — so `"*"` is acceptable here because
  `READY` carries no sensitive data).
- The sender treats the **first** `READY` whose `protocol`/`version` match (and `nonce` matches, if
  one was issued) as the signal to send the payload, and ignores later ones.

### 2. `HANDOFF_PAYLOAD` — sender → receiver

Sent by the local UI in response to `READY`, carrying the results.

```json
{
  "protocol": "rag-answer-batch-handoff",
  "version": 1,
  "type": "HANDOFF_PAYLOAD",
  "nonce": "<same nonce>",
  "payload": {
    "kind": "answer-batch-results",
    "source": "rag-cli-local-ui",
    "manifest_name": "Q3 vendor RFP",
    "knowledge_bases": ["vendor-docs", "policies"],
    "results_file": {
      "generated_at": "2026-07-15T10:00:00Z",
      "model": "stub-model",
      "results": [
        { "id": "1", "question": "What is your data retention policy?", "answer": "…" },
        { "id": "2", "question": "Do you support SSO?", "answer": "…" }
      ]
    }
  }
}
```

### 3. `HANDOFF_ACK` — receiver → sender (optional but recommended)

Sent after the receiver has parsed and accepted the payload, so the sender can show "Opened in Web
UI" and stop any retry loop.

```json
{
  "protocol": "rag-answer-batch-handoff",
  "version": 1,
  "type": "HANDOFF_ACK",
  "nonce": "<same nonce>"
}
```

### `HANDOFF_ERROR` — either direction (optional)

Sent when a well-formed protocol message could not be honored (e.g. receiver got a payload it
can't parse, or a version it doesn't support). Never send raw exceptions.

```json
{
  "protocol": "rag-answer-batch-handoff",
  "version": 1,
  "type": "HANDOFF_ERROR",
  "nonce": "<same nonce or empty>",
  "error": { "code": "UNSUPPORTED_VERSION | BAD_PAYLOAD | NONCE_CONSUMED | INTERNAL", "message": "human sentence" }
}
```

## Payload shape (`payload`)

The core `results_file` object is **byte-for-byte the local UI's Export JSON** and the CLI's
`BatchOutput` — reuse it, do not invent a variant:

| Field | Type | Required | Notes |
|---|---|---|---|
| `payload.kind` | `"answer-batch-results"` | yes | Lets the receiver route/validate; reject anything else. |
| `payload.source` | `"rag-cli-local-ui"` | yes | Provenance tag. |
| `payload.manifest_name` | string | no | Display label; may be absent for an ad-hoc run. |
| `payload.knowledge_bases` | string[] | no | KBs used, for display; `[]`/absent allowed. |
| `payload.results_file` | object | yes | The exported results file (below). |
| `payload.results_file.generated_at` | string (RFC 3339) | yes | May be `""` if unknown. |
| `payload.results_file.model` | string | yes | May be `""` if unknown. |
| `payload.results_file.results` | array | yes | One entry per Q&A. |
| `results_file.results[].id` | string | yes | |
| `results_file.results[].question` | string | yes | |
| `results_file.results[].answer` | string | yes | Empty string = unanswered; render distinctly, never as an error. |

Notes:
- The receiver MUST tolerate the `result` vs `results` key variance already handled in the local
  UI's `ui/lib/results.ts` (accept either; prefer `results`). Treat `results_file` as "the same
  thing a user could upload via the Web UI's import-a-results-file path."
- No per-question source/provenance is included — the batch results carry none today (documented
  gap). Do not design the receiver to require it.
- The payload is JSON-serializable only; `postMessage` uses structured clone, so no functions,
  DOM nodes, or class instances — plain objects/arrays/strings/numbers/booleans/null.

## Receiver responsibility after accepting a payload

Accepting a `HANDOFF_PAYLOAD` is **not** the end state. The Web UI MUST persist the results into
its own storage and then load them into view — holding the payload only in memory is not
sufficient (a refresh would lose it, and the user expects it to appear alongside their other
saved work).

1. **Persist to the Firebase `savedFiles` bank.** Once the user is authenticated, the receiver
   writes the accepted `results_file` (with its `manifest_name` / `knowledge_bases` metadata) into
   the Web UI's Firebase **`savedFiles`** collection as a new saved entry, exactly as if the user
   had imported the file themselves. This is the durable home for the handoff, not transient state.
2. **Gate on auth.** If the user is not yet authenticated when the payload arrives, the receiver
   holds the accepted payload in memory (the nonce is already consumed, so a stale tab cannot
   re-trigger it), prompts the user to sign in, and completes the `savedFiles` write immediately
   after authentication succeeds. The handoff MUST NOT be dropped just because auth was pending.
3. **Auto-load into view.** After the `savedFiles` write succeeds, the receiver automatically opens
   that saved entry in the results view so the user lands on the handed-off batch without a manual
   "open" step.
4. **Ack after persistence, not just receipt.** Send `HANDOFF_ACK` once the payload has been
   accepted (and, where practical, persisted) — so the local UI's confirmation reflects a durable
   handoff. If the `savedFiles` write fails, reply `HANDOFF_ERROR` with `INTERNAL` rather than a
   false `ACK`.

The exact Firebase schema of a `savedFiles` entry is the Web UI's own concern; the only contract
here is that the handed-off `results_file` becomes a first-class saved file and is then displayed.

## Origin validation

### Receiver (Web UI) validating the sender

The sender's origin is a **random loopback port**, so match by scheme + host, not an exact string:

- Accept the message only if **all** hold:
  - `event.origin` parses to scheme `http:` and hostname `127.0.0.1` **or** `localhost`
    (optionally also `[::1]`), any port.
  - `event.data.protocol === PROTOCOL` and `event.data.version === PROTOCOL_VERSION`.
  - `event.data.type === "HANDOFF_PAYLOAD"` and `payload.kind === "answer-batch-results"`.
- Reject/ignore silently otherwise (do not throw on unrelated messages — the window receives many).
- Keep a reference to `event.source` from the accepted message to send `ACK`/`ERROR` back.
- Do **not** trust any URL/onward-navigation strings in the payload as code or as a redirect
  target without validation; treat all payload strings as untrusted display data.

#### Nonce is single-use

- The `nonce` identifies **one** handoff and is **consumed on first accepted payload**. Once the
  receiver has accepted a `HANDOFF_PAYLOAD` for a given `nonce`, it MUST record that nonce as spent
  and **ignore any later message reusing it** — including a second `HANDOFF_PAYLOAD` with the same
  nonce.
- This prevents a stale open tab (still listening from an earlier handoff) from silently
  re-triggering an import, and makes a resent payload idempotent: the first wins, duplicates are
  dropped.
- A later message on a spent nonce SHOULD be ignored silently. If a reply is warranted (e.g. the
  same tab is deliberately retried), the receiver MAY answer `HANDOFF_ERROR` with a
  `NONCE_CONSUMED` code, but MUST NOT re-import.
- The spent-nonce record only needs to live as long as the receiver tab (in-memory is sufficient);
  it is a guard against re-processing within a session, not a persistent replay store.

### Sender (local UI) validating the receiver

The receiver's origin is fixed and known:

- Accept a `HANDOFF_READY`/`ACK`/`ERROR` only if:
  - `event.origin === WEB_UI_ORIGIN` (exact string match).
  - `event.data.protocol === PROTOCOL` and matching `version`.
  - `event.data.nonce === <the nonce issued for this handoff>` (when a nonce was issued).
- When **sending** `HANDOFF_PAYLOAD`, always pass `targetOrigin = WEB_UI_ORIGIN` (never `"*"`), so
  the results are delivered only to the intended app even if the tab navigated away.

## Knowing the receiver is loaded (no fixed timer)

- The sender captures the handle from `const win = window.open(url)` and does **not** post
  immediately.
- The sender installs a `message` listener first, then opens the tab, then waits for a validated
  `HANDOFF_READY` before posting the payload.
- **Retry/timeout:** if no `READY` arrives within `READY_TIMEOUT_MS` (suggest **10000 ms**), the
  sender may either (a) re-post nothing and show a "couldn't reach the Web UI — try Export JSON"
  fallback, or (b) as a belt-and-suspenders option, post the payload once at timeout targeted at
  `WEB_UI_ORIGIN` in case a `READY` was missed. Prefer (a); never loop-post.
- The receiver should send `READY` on mount **and** re-send once if it receives no payload within a
  few seconds (covers the case where the sender's listener attached late). Sending `READY` more
  than once is safe — the sender ignores duplicates by nonce.
- If `window.open` returns `null` (popup blocked), the sender falls back to the Export-JSON path and
  tells the user; there is no window to hand off to.

## URL the sender opens (step 1)

```
https://canonical-req-8605.web.app/import?handoff=1&nonce=<nonce>&v=1
```

- `handoff=1` tells the Web UI to enter receive-mode and emit `READY`.
- `nonce` is echoed in `READY` for correlation.
- `v` is the protocol version for an early mismatch check.
- The path (`/import`) is illustrative — the Web UI owns its route; only the query contract matters.
- The payload is **never** placed in the URL (size + it would leak into history/referrer).

## Non-goals / explicitly out of scope

- No fetch from the Web UI back into the daemon (the reason we use postMessage).
- No auth token crosses the boundary — the payload is data only.
- No streaming/partial handoff — one `HANDOFF_PAYLOAD` carries the complete result set.
- No persistence contract — if the Web UI wants a shareable link, that is its own backend concern,
  downstream of receiving the payload.

## Versioning

- `version` / `PROTOCOL_VERSION` is an integer bumped only on a breaking envelope/payload change.
- A receiver that gets a higher `version` than it supports SHOULD reply `HANDOFF_ERROR` with
  `UNSUPPORTED_VERSION` rather than silently mis-parsing.
- Additive, optional fields do **not** bump the version.
